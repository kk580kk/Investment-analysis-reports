# Xiaoyi GUI Agent

模拟真人在手机屏幕上的操作（点击、滑动、输入等），自动完成手机 App 内的各类任务。工具通过"截屏 → 理解画面 → 推理动作 → 执行操作"的循环工作，直到任务完成。

## 使用方法

调用 `xiaoyi-gui-tool` 工具，传入 `query` 参数。

## 参数

`xiaoyi-gui-tool` 只有一个参数：


| 参数      | 类型     | 必填  | 说明       |
| ------- | ------ | --- | -------- |
| `query` | string | 是   | 手机操作任务描述 |


**每次调用会重新启动 App 回到首页**，前一次调用的页面状态不会保留。因此同一 App 内有前后依赖的操作必须合并在一次 query 中完整描述。例如用户要求"在京东上找一款评价好的蓝牙耳机然后下单"，如果拆成先查商品、再下单两个 query，第二次调用会重启京东 App 回到首页，之前找到的商品页面已丢失，导致无法完成购买。正确做法是将搜索和下单写在同一个 query 中:“打开京东APP，搜索蓝牙耳机，按照好评排序，选择第一个商品下单购买。”。

## 构造 query

### 步骤

1. **理解意图**：从用户指令中提取核心目标，结合对话历史补全上下文。
2. **拆解任务**：如果用户指令包含多个子任务，以APP维度逐一拆解。单一任务可跳过此步。
3. **判断分发**：对每个子任务，先检查是否有专用工具（如天气工具、闹钟工具、日历工具等）可以更高效地完成。只有确认没有更合适的工具、且任务需要在 App 界面内操作时，才分发给本 Skill。能用专用工具完成的子任务优先用专用工具。
4. **补全信息**：用户原始指令往往缺少关键细节。先通过其他工具或对话获取缺失的上下文信息（如用户当前位置、具体日期等），再按下方补全指南补充 query 细节。
5. **组织指令**：按推荐结构输出 query。每个 query 只针对一个 App，涉及多个 App 的任务拆分为多个独立 query。

### 推荐结构

```
打开 <APP名称>，进入 <目标页面或功能>，执行 <具体操作>，并返回 <需要的信息或结果>。
```

### 应包含的要素


| 要素      | 必要性 | 说明              |
| ------- | --- | --------------- |
| 目标 APP  | 必填  | 要操作的 App 名称     |
| 具体操作    | 必填  | 需要执行的操作步骤或目标    |
| 期望结果    | 推荐  | 任务完成标准和需要返回的信息  |
| 数量/范围约束 | 推荐  | 查看几篇、搜索前几个等量化要求 |


### 补全指南

用户的原始指令通常不够具体，直接传给 GUI Agent 容易导致执行效率低下甚至失败。构造 query 时根据任务特点主动补全缺失信息：

**具体化模糊需求**：将"小红书看看美食推荐"这类模糊表达转化为明确操作，如"在小红书查看热度最高的 3 篇美食图文帖子并总结信息"。GUI Agent 需要清晰的操作目标才能高效执行。

**为探索性任务设定数量边界**：搜索和浏览类任务若不限定数量，GUI Agent 可能无限滚动，既耗时又容易出错。添加合理的数量约束，如"前 3 个"、"最热的 3 篇"。

**明确返回字段**：信息获取类任务需说明要返回必要字段（如"返回车次号、出发时间和票价"）。

**保持 query 精简**：query 只描述核心操作目标，不要塞入详细的分类框架、格式要求或总结模板。GUI Agent 会参考 query 中的每一项描述，内容越多冗余步骤越多，失败率越高。信息的归纳整理、格式化输出等后续加工交给外层完成，不要混入 query。

## 示例

### 信息查询

**用户：** "小红书找一下爆款装修方案总结到备忘录"

**good query：**

```
打开小红书，搜索"爆款装修方案"，浏览3篇热门图文笔记，并总结主要内容。
```

> 加入"图文"筛选和数量限制，query 简洁明确。"总结到备忘录"由其他工具处理，不混入 query。

**bad query：**

```
打开小红书APP，在搜索框输入"爆款装修方案"或"网红装修"进行搜索，浏览热门笔记，总结以下内容：1.流行的装修风格（如奶油风、极简风、原木风等）2.热门装修设计要点 3.装修预算分配建议 4.装修避坑经验 5.推荐的材料和家具。将总结好的内容完整告诉我，我会帮你保存到备忘录。
```

> query 过长且塞入大量预设分类，GUI Agent 会尝试逐一匹配这些条目，冗余步骤过多容易执行失败。只需给出核心目标，具体的信息整理交给外层完成。

---

**用户：** "帮我查一下明天北京到上海的高铁票价"

**query：**

```
打开铁路12306，搜索明天从北京到上海的高铁车次，返回前 3 个车次的车次号、出发时间和二等座票价
```

> 补充了数量限制（前 3 个）和具体返回字段，将模糊的"查票价"转化为明确的信息获取任务。

---

### 操作执行

**用户：** "在美团点一杯生椰拿铁"

**query：**

```
打开美团，进入外卖页面，搜索生椰拿铁，选择默认规格下单。
```

> 用户未指定规格，补充"默认规格"避免 GUI Agent 澄清。

---

**用户：** "相机间隔10s拍一次照片"

**query：**

```
打开相机，点击拍照按钮，然后结束任务。

```

> 用户希望间隔指定时间拍照，GUI Agent无法按照指定间隔执行任务，因此应该只将拍照query分发给GUI Agent。

### 内容发布类

**用户：** "帮我在小红书发布一篇关于春游的图文帖子"

**query：**

```
打开小红书APP，点击底部"+"号进入发布页面，选择相册中的春游照片，添加标题“贩卖春天”，内容写“趁着花期，和朋友们来一场说走就走的春游。草地、风筝、还有吃不完的零食，春天就该这么过呀！#春日 #春日美好时光 ”并发布帖子。
```

> 用户只给出主题和平台，未提供可直接发布的标题与正文。为满足"发布图文帖子"这一可执行目标，补全与春游主题匹配的标题和文案，减少来回澄清，让 GUI Agent 可一次完成发布。

### 复杂指令拆解

**用户：** "我明天要去北京出差，帮我查天气、查机票价格、查酒店价格、定闹钟并把行程添加到日历"

**拆解任务：** 拆出五个子任务——查天气、查机票、查酒店、定闹钟、添加日历。

**判断分发：** 查天气（天气工具）、定闹钟（闹钟工具）、添加日历（日历工具）均有专用工具，不经本 Skill。查机票和查酒店无专用工具、需要在 App 内操作，分发给本 Skill。

**补全信息：** 用户说"去北京"但未说出发地。"出差"通常指从当前城市出发，先通过定位工具获取当前城市（假设确认为上海），再补全到 query 中。

**组织指令：** 机票和酒店拆为两个独立 query：

**query 1（机票）：**

```
在携程APP查询明天上海到北京的机票，返回前3个航班的出发时间、到达时间和价格。
```

**query 2（酒店）：**

```
在携程APP查询明天北京市中心附近的酒店，返回酒店的名称、位置和价格。
```

## 输出说明

`xiaoyi-gui-tool` 返回文本结果，描述任务完成情况和获取到的信息。根据返回内容整理后呈现给用户即可。工具也可能返回失败信息（如遇到验证码等无法自动处理的情况），此时如实告知用户并建议手动操作。

**输出示例：**

```text
任务完成：已在小红书搜索"小艺claw"，以下是最新3篇帖子的总结：
1. 《小艺Claw体验报告》- 详细评测了小艺Claw的语音交互能力...
2. 《AI助手对比》- 对比了多款AI助手，小艺Claw在xxx方面表现突出...
3. 《新功能发布》- 小艺Claw最新更新了xxx功能...
```

```text
任务完成：已在拼多多完成今日签到，获得10积分。
```

```text
无法完成：拼多多签到页面需要滑动验证，当前无法自动完成，请手动操作。
```
